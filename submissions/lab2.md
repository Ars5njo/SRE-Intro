### 2.6 Service discovery answer

The gateway finds the events service through Docker Compose service discovery. All services are attached to the `app_default` bridge network, and Docker injects an embedded DNS resolver at `127.0.0.11` inside each container. The name `events` is registered as a DNS name for the events service on that network, so `http://events:8081` resolves without hardcoding an IP.

In the current run, `events` resolves from inside gateway to:

```text
172.18.0.5
```

The full observed reserve -> pay -> confirm span is about 764.7 ms, from the first events reserve log (`16:41:16.794483341`) to the final gateway `/reserve/{id}/pay` response (`16:41:17.559210883`). The ~746 ms gap between reserve response and payment is mostly client-side command overhead while parsing `reservation_id` and issuing the second `curl`, matching the lab's two-command flow.

For the pay/confirm request itself, the observable service-side span is about 11.1 ms, from the payments success log (`16:41:17.548125716`) to the final gateway response (`16:41:17.559210883`). Gateway does not log request-start timestamps, so this is the best measurement available from the existing logs.
